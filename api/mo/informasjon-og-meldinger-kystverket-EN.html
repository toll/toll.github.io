Information and Messages for Submission

Messages and Relationships Between Them

Message about Transport

For each port a vessel calls at, Norwegian Customs will have a transport message. This message shall contain information about which port the call concerns ("Port Of Call").

Message about 'Declaration of goods' (DOG)

All goods aboard the vessel shall be covered by a "Declaration of Goods" upon arrival at the port. This applies to both goods that are to be unloaded in the port the vessel arrives at and goods that are to be unloaded at a subsequent port.

For each port the vessel arrives at, new "Declaration of Goods" shall be created for all goods. All "Declaration of Goods" shall contain a code for which port the goods covered by "Declaration of Goods" are to be unloaded in â†’ "Port Of Discharge"

In the port where goods covered by "Declaration of Goods" are to be unloaded, the "Declaration of Goods" "Port Of Discharge" will be identical to the transport's (visit) "Port Of Call".

A "Declaration of Goods" can be submitted in different variants:

  1. A master consignment with all data and a list of house consignments with all data
  2. A master consignment with all data
    1. Underlying house consignment will in such cases be sent directly from system suppliers to Digitoll, and it is the consignment that will contain reference to the master consignment.
  3. A list of direct consignments to be unloaded in the same port, where all have all data
  4. A list of references. The references can be:
    1. A list of references to messages sent directly from system suppliers to Digitoll. There are two separate lists for master consignments and direct consignments respectively
    2. A list of references to previously submitted master consignments and/or a list of previously submitted direct consignments. There are two separate lists in the message.
    3. Common to both is that the references provided are waybill numbers.
      1. The lists can contain combinations of a and b (that is, one can simultaneously refer to messages sent directly to Digitoll with references to messages sent earlier from SSNN)

When a "Declaration of Goods" contains a master consignment or direct consignments, with all data, it shall not contain lists of references to previously submitted master consignments or direct consignments.

Message about 'Relationship'

The linking message, "Relationship", will contain the links between visitID and the various waybill references from consignments directly linked to the transport related to a port call.

  • Waybill numbers for ordinary consignments (House consignment), those that do not have a direct link to the transport, shall always be linked to a master consignment and through it to the transport. The ordinary consignments will therefore not be included in the relationship message sent at each port call.

The relationship message is used by Norwegian Customs to link the various submitted DOG messages with corresponding transport message.

Message about validation-status

For all submitted messages (transport, DOG and relationship), SSNN will receive synchronous response to submitted message. This will contain a requestID.

SafeSeaNet must use requestID to retrieve information about whether the submitted message has been processed OK in Digitoll. There will be specific endpoints for:

  • validation-status for transport
  • validation-status for DOG
  • validation-status for relationship (probably not shown in GUI in SSNN)

The message structure is common, requestID as parameter when sending message to Digitoll

The return structure is correspondingly similar, status, possibly MRN if OK, possibly error list if not OK.

  • Detailed description of message content in validation-status messages is documented in swagger, under query-messages: API documentation maritime-ssnn-status-api
  • General information about synchronous validation errors: Overview of validation errors for API in Notification/Information
  • General information about asynchronous validation errors: Overview of validation errors for API in MO

Display of validation-status results in SSNN screens

Response to transport submission and DOG submission respectively must be displayed in the screens in SSNN, so that one can also see the error message and what needs to be corrected.

In addition, it is important that the MRN returned from the transport message is displayed in the screens. The vessel may in some cases need to share this with the actors they collaborate with.
